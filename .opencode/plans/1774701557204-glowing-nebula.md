# Execution Plan: 15 Cross-Platform GUI Scripts with uv, Drag & Drop, and Middle Truncation

## Objective
Create 15 Python single-file scripts based on various GUI frameworks. Each script must be a self-contained `uv` executable (using PEP 723 inline metadata) that opens a window with specific features: resizable, title bar, multi-select list (with middle-truncation of file paths based on window width), 'Images' filter checkbox, Add/Remove/Clear/Print buttons, Drag & Drop support, and a status bar.

## Target GUI Frameworks
1. **wxPython**
2. **Tkinter + ttk + tkinterdnd2**
3. **Toga**
4. **PyWebView**
5. **PySide2** (Qt for Python)
6. **PySide6** (Qt for Python)
7. **PySimpleGUI**
8. **Dear PyGui**
9. **PyQt5**
10. **PyQt6**
11. **macOS: PyObjC** (Native Cocoa)
12. **macOS: Rubicon-ObjC** (Native Cocoa via Rubicon)
13. **Windows: pywin32** (Native Win32 API)
14. **Windows: winsdk** (Native Windows App SDK / WinRT)
15. **Windows: ctypes** (Raw Native Win32 API without pywin32)

## Global Script Requirements (PEP 723 & `uv`)
Every script must begin with a valid `uv` shebang and PEP 723 metadata block specifying dependencies.
```python
#!/usr/bin/env -S uv run
# /// script
# requires-python = ">=3.10"
# dependencies = [
#     "<framework-dependency>",
# ]
# ///
```

## Core Logic Components
To ensure consistency, each script should implement standard logic for:
1.  **Filtering:** `is_image(path)` checking for common extensions (`.png`, `.jpg`, `.jpeg`, `.gif`, `.bmp`, etc.).
2.  **State Management:** An internal list of absolute file paths to represent the "true" state. The UI list only shows the truncated representations.
3.  **Middle Truncation:** A function that recalculates the visible strings based on the current window/list width.
    *   *Implementation Strategy:* Due to the diverse frameworks, truncation must be handled dynamically on window resize events. It should measure the string width (using the framework's font metrics if available, or a fallback character count approximation if not) and insert an ellipsis (`⋮` or `...`) in the middle.

## Parallel Execution Strategy (Ultrawork)

The 15 scripts can be logically grouped into batches that can be executed in parallel by different subagents.

### Batch 1: The Qt Family (4 scripts)
**Targets:** PyQt5, PyQt6, PySide2, PySide6
**Agent Focus:** High similarity. Write one solid implementation (e.g., PySide6) and adapt imports and minor API changes for the others.
*   **Dependencies:** `PyQt5`, `PyQt6`, `PySide2`, `PySide6` respectively.
*   **DnD:** `setAcceptDrops(True)`, override `dragEnterEvent` and `dropEvent`. Extract URLs from `mimeData`.
*   **Truncation:** Use `QFontMetrics.elidedText` with `Qt.ElideMiddle` on the item delegates or during resize events.

### Batch 2: Standard & Cross-Platform Integrations (4 scripts)
**Targets:** Tkinter, wxPython, PySimpleGUI, Toga
**Agent Focus:** Standard python UI toolkits.
*   **Tkinter:** Needs `tkinterdnd2` for DnD. Uses `TkinterDnD.Tk()`. Bind `<<Drop>>`. Font measurement via `tkinter.font`.
*   **wxPython:** Needs custom `wx.FileDropTarget`. Override `OnDropFiles`. Text extent via `wx.Window.GetTextExtent`.
*   **PySimpleGUI:** Wraps Tkinter. Native DnD support might be limited; might need to inject standard Tkinter DnD bindings or use its built-in `enable_events` for drag and drop.
*   **Toga:** Uses native widgets. Needs to check Toga's current support for drag-and-drop and implement accordingly.

### Batch 3: Web-Based & Immediate Mode (2 scripts)
**Targets:** PyWebView, Dear PyGui
**Agent Focus:** Unconventional UI paradigms.
*   **PyWebView:** Requires HTML/JS frontend and Python backend. DnD handled via HTML5 Drag and Drop API in JS, passed to Python via JS API. Truncation can be handled via CSS (`text-overflow: ellipsis`, though middle ellipsis might require custom JS/CSS tricks or Python pre-processing).
*   **Dear PyGui:** Immediate mode GUI. Uses `dpg.add_drop_callback`. Truncation requires manual calculation as font metrics are handled differently.

### Batch 4: Native macOS (2 scripts)
**Targets:** PyObjC, Rubicon-ObjC
**Agent Focus:** Cocoa APIs via Python bridges.
*   **Dependencies:** `pyobjc-framework-Cocoa`, `rubicon-objc`.
*   **DnD:** Subclass `NSView`/`NSWindow`, register `NSPasteboardTypeFileURL`, implement `draggingEntered:` and `performDragOperation:`.
*   **Truncation:** Use `NSParagraphStyle` with `NSLineBreakByTruncatingMiddle` on `NSTextField` or `NSTableCellView`.

### Batch 5: Native Windows (3 scripts)
**Targets:** pywin32, winsdk, ctypes
**Agent Focus:** Win32/WinRT APIs.
*   **pywin32:** Use `win32gui.DragAcceptFiles`, intercept `WM_DROPFILES`, use `shell32.DragQueryFile`.
*   **ctypes:** Same concept as pywin32 but using raw `ctypes.windll.user32` and `ctypes.windll.shell32` calls. Requires defining necessary structs and constants manually.
*   **winsdk:** Use WinRT APIs (e.g., `Windows.ApplicationModel.DataTransfer`). Requires setting up a modern AppWindow.

## Atomic Commit Strategy
Each script will be developed independently and committed atomically.
*   **Commit Format:** `feat(<framework>): implement self-contained GUI script`
*   **Example:** `feat(PySide6): implement self-contained GUI script`
*   Each commit will contain exactly one `.py` file (e.g., `gui_pyside6.py`) ensuring zero cross-contamination.

## Development Workflow per Script
1.  **Skeleton & Metadata:** Create the file with the `uv` shebang and PEP 723 block.
2.  **UI Layout:** Construct the window, listbox, checkbox, buttons, and status bar.
3.  **State & Filtering:** Implement the file list state and the 'Images' filter logic.
4.  **Buttons:** Wire up the `+` (file dialog), `-` (remove), `X` (clear), and `V` (print) buttons.
5.  **Drag & Drop:** Implement framework-specific DnD logic to append to the internal state.
6.  **Middle Truncation:** Implement the resize event handler (or framework-specific feature) to visually truncate the displayed list items using the vertical ellipsis (⋮).
7.  **Status Bar:** Ensure the status bar updates whenever the list changes.
8.  **Test Execution:** Run `uv run <script>.py` to manually verify (or instruct the user to verify, since these are graphical scripts).
